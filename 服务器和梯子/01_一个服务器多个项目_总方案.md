# 一个服务器多个项目：总方案

> 目标：一台服务器同时跑 New API、Sub2API、CLIProxyAPI、你自己的 Web 项目、二创项目。互相不冲突，出问题也好排查。

## 1. 总体架构

```text
一台 Ubuntu 服务器
├── 宿主机 Caddy：负责 HTTPS 和反向代理
├── Docker：负责跑每个项目
├── 每个项目一个目录
├── 每个项目一个 docker-compose.yml
├── 每个项目一个独立端口
└── 每个项目自己管理自己的数据库/Redis/数据目录
```

不要把所有项目都塞到一个 compose 里。后期会乱，更新也麻烦。

正确方式：

```text
/opt/apps/newapi/docker-compose.yml
/opt/apps/sub2api/docker-compose.local.yml
/opt/apps/cliproxyapi/docker-compose.yml
/opt/apps/app1/docker-compose.yml
```

## 2. 服务器初始化

### 2.1 更新系统

```bash
sudo apt update && sudo apt upgrade -y
```

### 2.2 安装基础工具

```bash
sudo apt install -y curl wget git vim nano unzip tar htop ufw ca-certificates gnupg lsb-release
```

### 2.3 建项目总目录

```bash
sudo mkdir -p /opt/apps
sudo chown -R $USER:$USER /opt/apps
```

以后所有项目都放这里。

## 3. 安装 Docker 和 Compose

如果你已经装过 Docker，可以跳过。

```bash
curl -fsSL https://get.docker.com | sudo bash
sudo usermod -aG docker $USER
```

退出 SSH，重新登录服务器，然后检查：

```bash
docker version
docker compose version
```

能显示版本就行。

## 4. 安装 Caddy

```bash
sudo apt install -y debian-keyring debian-archive-keyring apt-transport-https
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/gpg.key' | sudo gpg --dearmor -o /usr/share/keyrings/caddy-stable-archive-keyring.gpg
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/debian.deb.txt' | sudo tee /etc/apt/sources.list.d/caddy-stable.list
sudo apt update
sudo apt install -y caddy
```

检查：

```bash
caddy version
sudo systemctl status caddy
```

## 5. 端口规划

建议固定一张表，别乱改。

| 项目 | 域名 | 宿主机端口 | 容器内端口 | 说明 |
|---|---|---:|---:|---|
| New API | `newapi.xcwindfall.top` | 3000 | 3000 | 模型中转管理平台 |
| Sub2API | `sub2api.xcwindfall.top` | 8080 | 8080 | 订阅转 API / 多账号管理 |
| CLIProxyAPI | `cliproxy.xcwindfall.top` | 8317 | 8317 | CLI 订阅转 API |
| 其他项目 1 | `app1.xcwindfall.top` | 18081 | 80/3000/8080 | 自己项目 |
| 其他项目 2 | `app2.xcwindfall.top` | 18082 | 80/3000/8080 | 二创项目 |

## 6. 端口绑定原则

对外项目端口建议这样写：

```yaml
ports:
  - "127.0.0.1:3000:3000"
```

意思是：

```text
只有服务器本机能访问 3000
公网用户不能直接访问 服务器IP:3000
公网用户只能通过 Caddy 的 https://newapi.xcwindfall.top 访问
```

这样更安全。

不建议写：

```yaml
ports:
  - "3000:3000"
```

这个会暴露到公网。

## 7. Docker 网络原则

每个项目 compose 里自己建一个 bridge 网络即可。

例如 New API：

```yaml
networks:
  newapi-network:
    driver: bridge
```

Sub2API：

```yaml
networks:
  sub2api-network:
    driver: bridge
```

不同项目之间默认互相隔离，减少冲突。

## 8. 数据目录原则

尽量用本地目录映射，不要只用 Docker named volume。

推荐：

```yaml
volumes:
  - ./data:/app/data
  - ./postgres_data:/var/lib/postgresql/data
  - ./redis_data:/data
```

好处：

```text
备份简单：tar 打包整个项目目录
迁移简单：scp 到新服务器再 docker compose up -d
排查简单：文件就在项目目录下面
```

## 9. 防火墙配置

查看：

```bash
sudo ufw status
```

配置：

```bash
sudo ufw allow 22/tcp
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw enable
```

如果你暂时需要直接访问某个端口调试，比如 3000：

```bash
sudo ufw allow 3000/tcp
```

调试完关闭：

```bash
sudo ufw delete allow 3000/tcp
```

## 10. 常用 Docker 命令

进入项目目录：

```bash
cd /opt/apps/newapi
```

启动：

```bash
docker compose up -d
```

停止：

```bash
docker compose down
```

重启：

```bash
docker compose restart
```

看日志：

```bash
docker compose logs -f
```

看某个服务日志：

```bash
docker compose logs -f newapi
```

看容器：

```bash
docker ps
```

看端口：

```bash
sudo ss -lntp
```

## 11. 多项目部署检查表

每新增一个项目，都检查这几个点：

```text
[ ] 是否单独建了 /opt/apps/项目名
[ ] 是否有独立 docker-compose.yml
[ ] 是否有独立 .env
[ ] 宿主机端口是否和其他项目冲突
[ ] 数据库/Redis 是否没有暴露公网
[ ] Caddyfile 是否加了新域名反代
[ ] DNS 是否解析到服务器 IP
[ ] docker compose up -d 是否成功
[ ] curl 127.0.0.1:端口 是否能访问
[ ] https://子域名 是否能访问
```
