# 修改其他项目端口和反代笔记

> 目的：让旧项目也适配“一个服务器多个项目”的方案，避免端口冲突，统一走 Caddy HTTPS。

## 1. 修改原则

每个项目只做三件事：

```text
1. 改 docker-compose.yml 的 ports
2. 改 Caddyfile 增加反代
3. 改域名 DNS 解析
```

## 2. 先查当前端口占用

```bash
sudo ss -tulpn
```

重点看：

```bash
sudo ss -tulpn | grep -E ':80|:443|:3000|:8080|:8081|:8082|:8317|:13000'
```

如果一个项目已经用了 80 或 443，必须改掉。  
80/443 只能给 Caddy。

## 3. 修改 docker-compose.yml

假设旧项目原来是：

```yaml
services:
  app:
    image: your-app:latest
    ports:
      - "80:80"
```

改成：

```yaml
services:
  app:
    image: your-app:latest
    ports:
      - "127.0.0.1:8081:80"
```

意思是：

```text
容器里面还是 80
宿主机改成 8081
并且只允许服务器本机访问
公网通过 Caddy 访问
```

## 4. 如果项目原来是 8080

原来：

```yaml
ports:
  - "8080:8080"
```

改成：

```yaml
ports:
  - "127.0.0.1:8081:8080"
```

如果 8081 被占用，就用 8082、8083、18080 都行。

## 5. 修改后重启项目

```bash
cd /opt/projects/你的项目目录
docker compose down
docker compose up -d
```

看日志：

```bash
docker compose logs -f
```

本机测试：

```bash
curl -I http://127.0.0.1:8081
```

## 6. Caddy 添加反代

编辑：

```bash
sudo nano /etc/caddy/Caddyfile
```

追加：

```caddyfile
app.xcwindfall.top {
    reverse_proxy 127.0.0.1:8081
}
```

检查并重载：

```bash
sudo caddy validate --config /etc/caddy/Caddyfile
sudo systemctl reload caddy
```

访问：

```text
https://app.xcwindfall.top
```

## 7. DNS 添加解析

域名后台添加：

| 主机记录 | 类型 | 值 |
|---|---|---|
| `app` | A | 你的服务器公网 IP |

如果你已经加了 `*` 泛解析，就不用每个子域名单独加。

## 8. 旧项目如果自带 Nginx 怎么办

没事。  
项目里面的 Nginx 可以继续保留，但它不要占宿主机 80/443。

例如：

```yaml
ports:
  - "127.0.0.1:8081:80"
```

然后 Caddy：

```caddyfile
app.xcwindfall.top {
    reverse_proxy 127.0.0.1:8081
}
```

也就是：

```text
公网 443
  ↓
宿主机 Caddy
  ↓
127.0.0.1:8081
  ↓
项目内部 Nginx:80
  ↓
项目应用
```

## 9. 旧项目如果已经有 Caddy 怎么办

建议删掉项目自己的 Caddy，只保留宿主机这一个 Caddy。

原因：

```text
多个 Caddy 抢 80/443 容易冲突
证书也不好统一管理
排查问题麻烦
```

最终只保留：

```text
/etc/caddy/Caddyfile
```

所有项目都写在这里。

## 10. 旧项目如果是裸进程，不是 Docker

比如项目启动在 5000：

```bash
python app.py --port 5000
```

Caddy 直接反代：

```caddyfile
oldapp.xcwindfall.top {
    reverse_proxy 127.0.0.1:5000
}
```

更推荐把旧项目也 Docker 化，方便长期维护。

## 11. 一个完整例子

### 旧项目 docker-compose.yml

```yaml
services:
  my-web:
    image: nginx:alpine
    container_name: my-web
    restart: always
    ports:
      - "127.0.0.1:8081:80"
    volumes:
      - ./html:/usr/share/nginx/html:ro
```

启动：

```bash
docker compose up -d
```

测试：

```bash
curl -I http://127.0.0.1:8081
```

### Caddyfile

```caddyfile
myweb.xcwindfall.top {
    reverse_proxy 127.0.0.1:8081
}
```

重载：

```bash
sudo caddy validate --config /etc/caddy/Caddyfile
sudo systemctl reload caddy
```

## 12. 排查清单

访问失败按这个顺序查：

```bash
# 1. 域名是否解析到服务器
nslookup app.xcwindfall.top

# 2. 项目容器是否启动
docker ps

# 3. 项目本机端口是否能访问
curl -I http://127.0.0.1:8081

# 4. Caddy 配置是否正确
sudo caddy validate --config /etc/caddy/Caddyfile

# 5. Caddy 是否正常
sudo systemctl status caddy --no-pager

# 6. Caddy 日志
journalctl -u caddy -f
```

## 13. 最终标准

每个项目都改成这样：

```text
项目容器：只绑定 127.0.0.1:项目端口
Caddy：负责公网 HTTPS
DNS：子域名指向服务器 IP
防火墙：只开放 22、80、443
```

这样后面你一个服务器部署 10 个、20 个项目都不乱。
